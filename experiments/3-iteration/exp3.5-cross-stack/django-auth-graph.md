The short-circuit on `PermissionDenied` enables any backend to act as a hard deny gate. rationale: unknown -- inferred from the backend iteration pattern.\n\n#### Compliance\n\n- Permission checks must always delegate to backends via the user model methods.\n- Never bypass the backend iteration to query permissions directly.\n- Superuser short-circuit: active superusers return `True` without backend queries.\n- `PermissionDenied` from any backend terminates iteration with `False`.\n- Permission strings must be in `app_label.codename` format.\n\n#### Participating Nodes\n\n`models`, `backends`, `decorators`, `mixins`, `__init__` (indirectly via `authenticate`).\n\n---\n\n### Aspect 3: `lazy-evaluation`\n\n```yaml\n# aspects/lazy-evaluation/aspect.yaml\nid: lazy-evaluation\nname: Lazy Evaluation\ndescription: >\n  User resolution on requests is deferred until first access using\n  SimpleLazyObject and caching patterns.\n```\n\n**Content: `aspects/lazy-evaluation/pattern.md`**\n\n#### What\n\nThe authentication system defers expensive operations (database queries, session reads) until their results are actually needed:\n\n1. **Request user resolution**: `AuthenticationMiddleware.process_request` sets `request.user` to a `SimpleLazyObject` wrapping `get_user(request)`. Each level adds or overrides behavior.\n\n#### Why\n\nApplications need to authenticate users from different sources (database, LDAP, OAuth, external web servers). pattern in mixins.\n\n#### Compliance\n\n- View protection must either redirect or deny, never silently pass.\n- Login URL resolution must fall back to `settings.LOGIN_URL`.\n- The `next` parameter
